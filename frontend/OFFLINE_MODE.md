# Офлайн-режим каталога профилей

Данный документ описывает функциональность офлайн-режима для страницы каталога профилей, которая позволяет пользователям просматривать каталог и изображения без активного подключения к интернету.

## Обзор

Офлайн-режим реализован с использованием Service Worker API и Cache API. Система автоматически кеширует данные каталога и изображения профилей, обеспечивая доступность контента даже при отсутствии сетевого подключения.

### Основные возможности

- **Автоматическое кеширование**: Данные и изображения сохраняются локально при первой загрузке
- **Индикация статуса**: Визуальные индикаторы показывают состояние подключения и время последней синхронизации
- **Принудительное обновление**: Возможность вручную обновить данные из сети
- **Умное управление кешем**: Автоматическое удаление старых записей при достижении лимита (LRU стратегия)
- **Graceful degradation**: Приложение продолжает работать при недоступности Service Worker

## Стратегии кеширования

Система использует две различные стратегии кеширования в зависимости от типа ресурса:

### Network First (для данных каталога)

**Применяется к**: `/api/catalog/*`

**Логика работы**:
1. Сначала пытается получить данные из сети (таймаут 5 секунд)
2. При успехе обновляет кеш и возвращает свежие данные
3. При неудаче возвращает данные из кеша (если доступны)

**Преимущества**: Приоритет актуальности данных, но с fallback на кеш

```javascript
// Псевдокод
async function networkFirst(request) {
  try {
    const response = await fetchWithTimeout(request, 5000)
    await cache.put(request, response.clone())
    return response  // Свежие данные
  } catch {
    return await cache.match(request)  // Кешированные данные
  }
}
```

### Cache First (для изображений)

**Применяется к**: `/static/photos/*`, `/static/thumbnails/*`

**Логика работы**:
1. Сначала проверяет наличие в кеше
2. Если найдено, возвращает из кеша (быстро)
3. Если не найдено, загружает из сети и сохраняет в кеш
4. При ошибке возвращает placeholder изображение

**Преимущества**: Мгновенная загрузка изображений, экономия трафика

```javascript
// Псевдокод
async function cacheFirst(request) {
  const cached = await cache.match(request)
  if (cached) {
    return cached  // Из кеша
  }
  
  try {
    const response = await fetch(request)
    await cache.put(request, response.clone())
    return response  // Из сети
  } catch {
    return await cache.match('/placeholder-image.svg')
  }
}
```

### Управление размером кеша

- **Лимит изображений**: 500 изображений
- **Стратегия удаления**: LRU (Least Recently Used)
- **Автоматическая очистка**: При достижении лимита удаляется самое старое изображение
- **Метаданные**: Время последнего использования хранится в IndexedDB

## Тестирование офлайн режима

### Метод 1: Chrome DevTools (рекомендуется)

1. Откройте приложение в браузере
2. Откройте Chrome DevTools (F12)
3. Перейдите на вкладку **Network**
4. В выпадающем списке **Throttling** выберите **Offline**
5. Обновите страницу или перейдите на страницу каталога

**Ожидаемое поведение**:
- Индикатор статуса показывает "Офлайн"
- Данные каталога загружаются из кеша
- Изображения отображаются из кеша
- Кнопки создания/редактирования профилей отключены

### Метод 2: Системное отключение сети

1. Откройте приложение и загрузите страницу каталога (чтобы данные закешировались)
2. Отключите Wi-Fi или Ethernet на компьютере
3. Обновите страницу или перейдите на каталог

### Метод 3: Service Worker Override

В консоли DevTools выполните:

```javascript
// Симуляция офлайн режима
navigator.serviceWorker.controller.postMessage({ type: 'SIMULATE_OFFLINE' })
```

### Проверка кеша

Для просмотра содержимого кеша:

1. DevTools → **Application** → **Cache Storage**
2. Разверните домен приложения
3. Вы увидите три кеша:
   - `catalog-data-v1` - данные каталога
   - `catalog-images-v1` - изображения профилей
   - `static-assets-v1` - статические ресурсы

### Проверка IndexedDB

Для просмотра метаданных:

1. DevTools → **Application** → **IndexedDB**
2. Разверните `catalog-offline`
3. Проверьте stores:
   - `metadata` - содержит `lastSync` timestamp
   - `lru` - содержит записи с `url` и `lastUsed`

## UI компоненты

### OfflineStatus

Компонент отображается в верхней части страницы каталога и показывает:

- **Индикатор подключения**: 
  - 🟢 "Онлайн" - активное подключение
  - 🟡 "Офлайн" - работа без сети
  
- **Время синхронизации**: 
  - "Обновлено: только что"
  - "Обновлено: 5 мин назад"
  - "Обновлено: 2 ч назад"
  - "Данные не синхронизированы"

- **Кнопка обновления**: 
  - Принудительно загружает свежие данные из сети
  - Показывает индикатор загрузки во время обновления
  - Отключена в офлайн режиме

### Отключение функций в офлайн режиме

При отсутствии подключения автоматически отключаются:
- Кнопка создания нового профиля
- Кнопки редактирования существующих профилей
- Кнопка удаления профилей

При наведении на отключенные кнопки отображается tooltip: "Недоступно в офлайн режиме"

### Версионирование кешей

Система использует версионированные имена кешей для безопасного обновления:

```javascript
const VERSION = 'v1'
const CACHE_NAMES = {
  catalog: `catalog-data-${VERSION}`,
  images: `catalog-images-${VERSION}`,
  static: `static-assets-${VERSION}`
}
```

При обновлении версии:
1. Новый Service Worker создает кеши с новыми именами
2. При активации удаляются кеши предыдущих версий
3. Пользователи получают свежие данные

## Обработка ошибок

### QuotaExceededError

При превышении лимита хранилища:
1. Автоматически удаляется 20% самых старых записей
2. Операция кеширования повторяется
3. При повторной ошибке операция пропускается (приложение продолжает работать)

### Недоступность Cache API

Если Cache API недоступен (старые браузеры):
- Service Worker работает в режиме "только сеть"
- Все запросы проходят напрямую без кеширования
- Приложение продолжает нормально функционировать

### Таймаут сети

Для запросов данных установлен таймаут 5 секунд:
- При превышении таймаута используется кеш
- Предотвращает долгое ожидание при медленном соединении

### Очистка кеша

Для полной очистки кеша и IndexedDB:

```javascript
// В консоли браузера
async function clearAllCache() {
  // Удалить все кеши
  const cacheNames = await caches.keys()
  await Promise.all(cacheNames.map(name => caches.delete(name)))
  
  // Удалить IndexedDB
  indexedDB.deleteDatabase('catalog-offline')
  
  console.log('All cache cleared')
}

clearAllCache()
```

Или через DevTools:
1. **Application** → **Clear storage**
2. Выберите **Cache storage** и **IndexedDB**
3. Нажмите **Clear site data**

### Принудительное обновление Service Worker

Для немедленного обновления Service Worker:

1. DevTools → **Application** → **Service Workers**
2. Нажмите **Update**
3. Или установите флаг **Update on reload**

## Производительность

### Метрики

- **Первая загрузка**: Данные загружаются из сети (~200-500ms)
- **Повторная загрузка (онлайн)**: Данные из сети с обновлением кеша (~200-500ms)
- **Загрузка офлайн**: Данные из кеша (~10-50ms) - **в 10-20 раз быстрее**
- **Изображения (кеш)**: Мгновенная загрузка (~5-10ms)

### Использование хранилища

- **Данные каталога**: ~50-200 KB (зависит от количества профилей)
- **Изображения**: ~500 KB - 2 MB на изображение
- **Максимальный размер**: ~1 GB (лимит 500 изображений × ~2 MB)
- **Метаданные**: ~10-50 KB (IndexedDB)

## Часто задаваемые вопросы

### Как узнать, что данные загружены из кеша?

Проверьте индикатор статуса в верхней части страницы каталога. Если показано "Офлайн" или время последней синхронизации не "только что", данные загружены из кеша.

### Как часто обновляются данные?

Данные обновляются автоматически при каждом посещении страницы каталога (если есть подключение). Также можно принудительно обновить, нажав кнопку обновления.

### Что происходит при изменении данных на сервере?

При следующем посещении страницы (онлайн) данные автоматически обновятся. Кеш всегда обновляется при успешном сетевом запросе.

### Как работает Service Worker?

Service Worker регистрируется автоматически при загрузке приложения, если браузер поддерживает эту технологию. Никаких дополнительных настроек не требуется.

### Что делать, если кеш поврежден?

Очистите кеш через DevTools (см. раздел "Очистка кеша") или выполните hard refresh (Ctrl+Shift+R / Cmd+Shift+R).
